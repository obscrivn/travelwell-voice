# Agent Spec

## Overview
TravelWell AI is a travel wellness concierge agent that extracts user travel contexts and wellness needs, researches local fitness facilities, filters based on accessibility and pricing, ranks the top 3 recommendations, and returns a detailed itinerary and progress trace. 

The implementation utilizes a hybrid model:
* **Backend (`backend/`):** Implements one ADK orchestration workflow coordinating two specialized LLM agents (`Research & Intelligence Agent` and `Ranking & Itinerary Agent`).
* **Frontend (`frontend/`):** TypeScript + React displaying map locations, recommendations, and logical traces.

---

## Design Philosophy
TravelWell AI demonstrates how specialized AI agents, deterministic services, and structured workflows collaborate to solve a practical real-world travel problem while remaining modular, explainable, and extensible.

---

## Current MVP Scope
This version focuses exclusively on gyms, recreation centers, indoor pools, and related physical fitness/wellness facilities. Other wellness experiences (such as healthy restaurants, hiking trails, pickleball courts, and yoga studios) are designated as future version enhancements.

---

## Recommendation Philosophy
Recommendations must satisfy all mandatory user constraints (e.g., budget caps, required amenities like showers) before optimizing for preferred amenities (e.g., indoor pools, specific memberships) and travel convenience.

---

## Workflow Transparency
The frontend displays logical workflow stages (e.g., Trip Context, Fitness Discovery, Access & Membership, Facility Intelligence, Ranking, Itinerary, and Visualization) rather than physical ADK agents. This decouples the user experience from the backend execution topology, allowing backend orchestration code to evolve and scale without requiring changes to the user interface.

---

## Example Use Cases
* **Input Prompt:** "I'll be in downtown Chicago tomorrow from 6-9 PM. I have a YMCA membership, but willing to pay up to $20. I prefer an indoor pool or treadmill, need showers."
* **Expected Output:**
  * Top Recommended Option: Downtown Chicago YMCA (Reason: 100% matched amenities, $0 cost via reciprocity, nearby).
  * 2nd Recommended Option: FFC Union Station (Reason: premium saline pool, fits $20 day pass budget limit).
  * Structured timeline itinerary with walking durations.
  * Active trace illustrating logical steps (Trip Context ➔ Fitness Discovery ➔ Access & Membership ➔ Facility Intelligence ➔ Ranking ➔ Itinerary).

---

## Tools Required
1. **Google Places Discovery (`places_service`):** For geolocation queries and finding candidate locations.
2. **Access & Amenity Scraper (`scraper_service`):** Fetches or simulates fetching facility guest policies, pricing, and pool schedules.
3. **Google Maps Routing (`routing_service`):** Computes distance, transit options, and duration buffers between the traveler's coordinates and candidates.
4. **Mock Data Provider (`mock_data_service`):** Deterministic mock data fallback when credentials or network is unavailable (specifically targeting the Chicago Loop baseline scenario in `tests/mock_data.json`).

---

## Constraints & Safety Rules
* **No Real-Time Guarantees:** Do not make hard claims about real-time slot availability or open pool lanes unless scraped explicitly. Show clear indicator tags for estimated statuses.
* **No Automatic Bookings:** The system must never attempt to book, reserve, or purchase passes.
* **Privacy Boundary:** Trip locations and user profiles must only be stored per-session in memory (resumed via memory bank or local session ID). No long-term storage of user telemetry.

---

## Success Criteria
* **Correct Parsing:** Extraction of trip window, location, budget, preferred amenities, and membership affiliations with >95% accuracy in evals.
* **Optimal Selection:** The recommended top 1 option must be the most cost-efficient (e.g. leveraging memberships) and closest facility satisfying all strict constraints.
* **Trace Accuracy:** The trace log must successfully reflect all workflow events step-by-step.
* **Offline Robustness:** When `use_mock_data` is `true`, the pipeline must run 100% deterministically without network calls.
